## Project Overview

**babylon-mac-local** is a local development orchestration framework for running a complete Babylon blockchain testnet on macOS. It manages 13+ services as git submodules, including Babylon nodes, Bitcoin regtest network, vigilante services, finality providers, BTC staking, and IBC relayer.

**Primary Language**: Shell scripting (Bash)
**Build System**: go-task (Taskfile.yaml) + Make
**Process Management**: Supervisord
**Target Components**: Go-based blockchain services (built from submodules)

## Essential Commands

### Initial Setup
```bash
# Pull all submodule components
git submodule update --init --recursive

# Build all 7 components (babylon, btc-staker, covenant-emulator, finality-gadget, finality-provider, vigilante)
task build-all

# Initialize network configuration (creates testnets/ directory with configs)
task init-network-conf

# Start all services via supervisord
task up-babylon-network

# Initialize Bitcoin wallets and generate blocks
task init-bitcoin
```

### Service Management
```bash
# Check status of all services
task ps

# Stop services and remove all data
task clean

# Remove testnet data only (without stopping services)
task rm-all

# Start/stop supervisord manually
task up-supervisord
task down-supervisord
```

### IBC/Relayer Setup (Sequential)
```bash
# 1. Deploy Babylon contracts to consumer chain
task deploy_babylon_contract

# 2. Initialize IBC relayer configuration
task init-relayer

# 3. Start cross-chain relaying
task start-relayer
```

### Development
```bash
# List all available tasks
task list

# View supervisord web interface
open http://127.0.0.1:39001
```

## Architecture

### Service Topology

**Babylon Blockchain** (2 nodes):
- node0: RPC 26657, P2P 26656
- node1: RPC 26667, P2P 26666

**Bitcoin Integration**:
- bitcoindsim: Regtest network (RPC 18443, ZMQ 29000-29002)
- vigilante: 4 roles monitoring Bitcoin ↔ Babylon
  - reporter (8080): Reports Bitcoin data to Babylon
  - submitter (8081): Submits checkpoints to Bitcoin
  - monitor (8082): Monitors checkpoint finalization
  - bstracker (8083): Tracks BTC staking transactions

**Finality & Staking**:
- finality-provider (12581): Main finality provider
- consumer-fp (12582): Consumer chain finality provider
- eotsmanager (15813): EOTS signature manager
- consumer-eotsmanager (15814): Consumer EOTS manager
- btc-staker (15812): BTC staking service
- covenant-emulator (4116): Covenant signature emulation

**Cross-Chain**:
- ibcsim-bcd (36657): IBC consumer chain
- relayer: Cosmos IBC relayer (configured via init-relayer)

### Configuration System

**Base Config**: `conf/baseconfig.yaml`
- Defines all service ports and endpoints
- Directory structure for runtime data
- Service command definitions

**Runtime Data**: `testnets/` (gitignored)
- Generated by `task init-network-conf`
- Contains per-service configs, keyrings, and data
- Logs centralized in `testnets/logs/`

**Path Substitution**: `lib/utils.sh` contains `CODE_BASE` variable that MUST be updated to your local path before running init-network-conf.

### Key Orchestration Functions

Located in `lib/utils.sh`:
- `init_config()`: Initializes all service configurations
- `bitcoin_init()`: Creates Bitcoin wallets and generates blocks
- `deploy_babylon_contract()`: Deploys contracts to consumer chain
- `init_relayer()`: Configures IBC relayer
- `start_relayer()`: Starts cross-chain relaying

## Critical Configuration Requirements

1. **Update CODE_BASE**: Before first run, edit `lib/utils.sh:3` to set `CODE_BASE` to your local repository path.

2. **Supervisord Version**: Must use 4.2.5 (specified in README)

3. **Bitcoin Version**: Must use v28.0.0

4. **Go Version**: Use 1.23.3 (managed via gvm)

5. **Sequential Workflow**: Services must be started in order:
   - Build all → Init config → Start network → Init Bitcoin → Deploy contracts → Init relayer → Start relayer

## Submodule Structure

All blockchain components are git submodules:
- `babylon/` - Main Babylon blockchain
- `btc-staker/` - BTC staking service
- `covenant-emulator/` - Covenant emulator
- `finality-gadget/` - Finality mechanism
- `finality-provider/` - Finality provider
- `vigilante/` - Bitcoin monitoring
- `babylon-sdk/` - SDK and contracts
- `relayer/` - IBC relayer
- `optimism/` - Optimism integration
- `ibcsim-bcd/` - IBC consumer chain

Each submodule builds independently using Make. The orchestration layer (this repo) only manages configuration and process lifecycle.

## Common Issues

1. **Services won't start**: Check `CODE_BASE` in `lib/utils.sh` matches your local path
2. **Port conflicts**: All ports defined in `conf/baseconfig.yaml` must be available
3. **Build failures**: Ensure Go 1.23.3 is active (`gvm use go1.23.3`)
4. **Relayer issues**: Must run `deploy_babylon_contract` before `init-relayer`
5. **Bitcoin not responding**: Run `task init-bitcoin` after network is up

## Development Notes

- **Immutable Config Pattern**: Template configs in `conf/` are copied to `testnets/` during init
- **No Tests**: This orchestration layer has no test suite; testing happens in submodules
- **Chinese Documentation**: Primary README is in Chinese
- **Metrics**: Prometheus metrics available on multiple services (ports 2112-2116, 4112-4116)
- **Keyring Backend**: All services use `--keyring-backend=test` for development
